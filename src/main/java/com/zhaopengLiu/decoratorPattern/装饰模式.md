**装饰模式**
       
    用组合代替继承将抽象和具体实现分离开来，实现解耦合
    抽象和实现分离，使其可各自独立变化
    
**优点**
    
    装饰模式相对于继承模式比较灵活
    装饰模式可以设计多个具体的装饰类，创造多个不同行为的组合
   
**缺点**

    装饰模式过度使用造成很多子类，增加系统的复杂度

**结构**
    
    客户端 --> 抽象构件 -->  具体构件
               ^           
               |           
               |           
              抽象装饰 -->  具体装饰
            
例： BufferReader


应用场景：
     
     当需要一个给一个类拓展职责的时候，不能用子类方法拓展，该类被隐藏，或者修改该类的职责会产生大量的子类；
     需要给一个类附加一些需要排列的方法时，继承方式很难实现
     当类的职责可以动态的添加也可以动态的取消时
 
拓展：
      
     当类的结构并不复杂的时候可以只保留抽象构件接口或者抽象装饰接口
